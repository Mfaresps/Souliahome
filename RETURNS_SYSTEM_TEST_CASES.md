## ✅ سيناريوهات الاختبار الناجحة

### TC-001: استرجاع بسيط ناجح
```
المتطلبات:
- فاتورة مكتملة بـ 150 ج
- ضمن فترة 14 يوم
- خزنة (كاش) بها رصيد

الخطوات:
1. POST /returns
   {
     "originalTransactionId": "tx_001",
     "items": [{"code": "SKU-001", "name": "منتج", "qty": 1, "price": 150, "total": 150}],
     "requestKind": "return",
     "reason": "تلف الشحنة"
   }

2. POST /returns/{id}/approve

النتيجة المتوقعة:
✅ Status: 201 Created
✅ Return status: "معلق" (pending)
✅ معاملة مرتجع تُنشأ تلقائياً عند الموافقة
✅ خزنة تنقص ب 150 ج
✅ مخزن: SKU-001 يزيد ب 1
```

## ❌ سيناريوهات الرفض

### TC-005: خارج فترة الـ 14 يوم
```
الشرط: الفاتورة من 20 يوم ماضي

النتيجة المتوقعة:
❌ HTTP 400 Bad Request
❌ Message: "فترة الاسترجاع انتهت (20 يوم ماضي)"
❌ الطلب يُرفض ولا يُنشأ
```

### TC-006: أصناف غير موجودة في الفاتورة الأصلية
```
الشرط: العميل يطلب استرجاع SKU-999 الذي لم يكن في الفاتورة

النتيجة المتوقعة:
❌ HTTP 409 Conflict
❌ Message: "الصنف SKU-999 غير موجود في الفاتورة الأصلية"
❌ الطلب مُرفوض + معايرة من الإداري مطلوبة
```

### TC-007: استرجاع مكرر نفس العناصر
```
الحالة:
1. طلب استرجاع رقم 1: SKU-001 → معتمد
2. محاولة طلب استرجاع رقم 2: SKU-001 نفسها

النتيجة المتوقعة:
❌ HTTP 409 Conflict
❌ Message: "SKU-001 تم استرجاعها مسبقاً (ID: return_001)"
❌ الطلب الثاني مُرفوض
```

### TC-008: خزنة غير كافية للرد
```
الشرط:
- طلب رجوع: 500 ج
- رصيد (كاش): 200 ج فقط

النتيجة المتوقعة:
❌ HTTP 400 Bad Request (عند الموافقة)
❌ Message: "رصيد الخزنة (كاش) غير كافٍ - المتاح: 200، المطلوب: 500"
❌ الموافقة ترفع + طلب إضافة رصيد
```

### TC-009: المنتج البديل غير متاح
```
الشرط:
- طلب استبدال SKU-002
- الكمية المتاحة: 0

النتيجة المتوقعة:
❌ HTTP 400 Bad Request
❌ Message: "الكمية المتاحة من SKU-002: 0، المطلوب: 1"
⚠️ الطلب يُحفظ لكن status: "pending_inventory"
```

### TC-010: فاتورة ملغاة
```
الشرط: الفاتورة الأصلية مع `cancelled: true`

النتيجة المتوقعة:
❌ HTTP 400 Bad Request
❌ Message: "لا يمكن استرجاع من فاتورة ملغاة"
```

## 🔐 اختبارات الأمان

### TC-011: محاولة إنشاء طلب بـ 3 معاملات منفصلة لنفس البيانات
```
السيناريو الاحتيالي:
- إنشاء طلب استرجاع A → معتمد
- إنشاء طلب استرجاع B (نفس البيانات) → معتمد
- إنشاء طلب استرجاع C (نفس البيانات) → معتمد

النتيجة المتوقعة:
✅ الطلب A: معتمد + معاملات
❌ الطلب B: مرفوض - "تم استرجاع هذه الأصناف مسبقاً"
❌ الطلب C: مرفوض - "تم استرجاع هذه الأصناف مسبقاً"
```

```
النتيجة المتوقعة:

❌ Message: "سعر SKU-001 مختلف - كان 500، الآن 50"
```

## 📊 اختبارات التكامل (Integration Tests)

### TC-014: تتبع المعاملات الثلاثية

```javascript
السيناريو: استرجاع 150 ج

خطوات التحقق:

1️⃣ المخزن (Inventory)
   SELECT * FROM transactions 
   WHERE linkedReturnId = 'return_001'
   AND type = 'مرتجع'
   
   ✅ النتيجة:
   {
     id: "tx_return_001",
     type: "مرتجع",
     ref: "INV-2025-0001-RET",
     items: [{code: "SKU-001", qty: 1}],
     linkedReturnId: "return_001"
   }

2️⃣ السجلات (Transactions)
   SELECT * FROM transactions
   WHERE ref = 'INV-2025-0001-RET'
   
   ✅ يجب أن توجد معاملة واحدة فقط

3️⃣ الخزنة (Vault)
   SELECT balance FROM vault 
   WHERE segment = 'كاش'
   
   ✅ يجب أن تنقص ب 150
   
4️⃣ التدقيق (Audit)
   SELECT * FROM expenses
   WHERE linkedTransactionId = 'tx_return_001'
   
   ✅ يجب أن توجد expense واحدة بـ amount: -150
```

### TC-015: حالة Race Condition
```
السيناريو: طلبان متزامنان لاسترجاع نفس العنصر

Timeline:
T1: POST /returns (SKU-001) → جاري التحقق
T2: POST /returns (SKU-001) → جاري التحقق
T1: ← معتمد ✅
T2: ← ???

النتيجة المتوقعة:
T1: معتمد + معاملات
T2: مرفوض - "SKU-001 تم استرجاعها مسبقاً"

التحقق:
- T2 يجب أن يشهد التغيير من T1 قبل الموافقة
- استخدام MongoDB transaction locks
```

```
- SKU-003: 300 ج

✅ معاملة رجوع: 2 أصناف

✅ Debt: 50 ج
```
